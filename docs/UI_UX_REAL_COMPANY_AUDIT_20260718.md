# Agent Company OS 실제 회사 UI/UX 전수 감사

감사일: 2026-07-18  
대상: `UI 전체흐름 검증 회사`와 완료 목표 `UI 전체완료 사용자 여정 검증`  
환경: 실행 중인 로컬 서버 `http://127.0.0.1:5173`, 데스크톱 1440×1000, 모바일 390×844

## 1. 결론

핵심 업무 흐름과 조회 기능은 동작하지만, 현재 제품은 “실제 회사의 업무 운영 화면”보다 “구현 상태를 직접 살펴보는 기술 운영 콘솔”에 가깝다. 가장 큰 문제는 시각 장식이 아니라 동일한 완료 목표의 상태가 페이지마다 다르게 표현되는 점이다. 오너는 무엇이 완료됐고 무엇을 조치해야 하는지 하나의 진실로 판단할 수 있어야 한다.

종합 평가는 5점 만점 기준 2.4점이다.

| 평가 축 | 점수 | 판단 |
|---|---:|---|
| 상태 신뢰성과 업무 판단 | 1.5 | 완료·active·취소·승인 대기·복구 중이 동시에 표시됨 |
| 정보 구조와 탐색 | 2.5 | 주요 기능은 찾을 수 있으나 페이지 역할이 중복되고 기술 용어가 전면에 노출됨 |
| 작업 효율 | 2.5 | 조회·선택·탭 전환은 동작하지만 기본 선택과 다음 행동 안내가 약함 |
| 피드백·로딩·오류 복구 | 2.0 | 일부 첫 진입에서 로딩 표시 없이 빈 화면이 나타남 |
| 시각적 계층과 일관성 | 2.0 | 어두운 앱 셸과 흰 업무 화면이 충돌하며 목표 보드의 대비가 심각하게 낮음 |
| 모바일 사용성 | 3.0 | 문서 전체 가로 오버플로는 없지만 잘림·밀집·작은 터치 영역이 많음 |
| 접근성 | 2.0 | 한 페이지 한 H1과 랜드마크는 좋지만 작은 글자, 작은 조작 영역, 이름 없는 입력 요소가 존재함 |
| 실제 회사 오너 준비도 | 2.0 | 기술 ID와 내부 상태를 해석해야 하며 “오늘 결정할 것” 중심 화면이 없음 |

## 2. 확인 범위와 증거

- 로그인과 로그인 비활성 상태
- 내 회사, 회사 홈, 회사 목표, 직원, 회의, 통합 검색·알림
- 프로젝트 워룸, 실행 워크스페이스, 픽셀 오피스
- 운영 플랫폼, 운영 상태 센터
- 목표 보드·타임라인, 직원 상세, 프로젝트 Task 상세
- 보고·검색 결과·회사 안전 관리, 업무 흐름·산업 템플릿
- 운영 이벤트 스트림 연결, 모바일 사이드 메뉴

실제 UI 조작으로 목표 보드 6열, 타임라인 87건, 직원 3명, 프로젝트 Task 6개, 회의 6개, 보고 233건을 확인했다. 운영 이벤트 스트림은 `연결됨` 상태로 바뀌었다. 앱 콘솔 오류, 페이지 예외, HTTP 4xx/5xx는 없었고 React Router 차기 버전 안내 경고 두 종류만 반복됐다.

## 3. 우선 해결해야 할 문제

### P0-1. 동일 업무의 상태가 페이지마다 모순된다

확인된 표현은 다음과 같다.

- 회사 목표: 완료, 100%, Task 5/5, 승인 대기 0
- 프로젝트 워룸: 프로젝트 `active`, 진행 6/6
- 실행 워크스페이스: 취소된 Run이 기본 선택되고 결과 승인 `대기 중`
- 검색·알림: 읽지 않은 warning 11건, 여러 `승인 대기`
- 회사 안전 관리: pending approval 1건, draft meeting summary 1건
- 픽셀 오피스: 첫 진입 시 `상태 복구 중`
- 운영 플랫폼: 완료율 100%와 동시에 오래된 결과물 15건을 `위험`으로 표시
- 운영 상태: 서비스는 정상이나 이벤트 스트림은 사용자가 연결하기 전 `연결 안 됨`

개선 방향:

1. Goal, Delivery, Project, Run, Approval, Notification의 표시 상태를 하나의 프레젠테이션 모델에서 계산한다.
2. 재시도·수정 요청으로 대체된 Run은 기본 목록에서 `과거 시도`로 접고 현재 유효 Run을 먼저 선택한다.
3. 목표 완료 시 이미 해소된 승인 알림을 자동 해결하고 프로젝트 상태도 완료로 정합화한다.
4. 모든 상태 카드에 `왜 이 상태인지`와 `사용자가 할 다음 행동`을 함께 제공한다.

완료 기준: 동일 목표를 어느 페이지에서 열어도 현재 유효 상태, 남은 승인 수, 완료율이 일치해야 한다.

### P0-2. 목표 보드와 타임라인의 색 대비가 읽기 불가능한 수준이다

목표 상세는 별도의 밝은 테마를 사용하지만 전역 어두운 테마의 글자색이 일부 남아 있다. 보드 열 제목과 타임라인 이벤트명이 흰색 또는 매우 옅은 색으로 렌더링돼 밝은 배경에서 거의 보이지 않는다.

개선 방향:

- 밝은 목표 영역 안의 모든 텍스트·링크·상태 토큰을 명시적인 light-surface 토큰으로 통일한다.
- WCAG AA 기준으로 일반 텍스트 4.5:1, 큰 텍스트 3:1 이상을 자동 검사한다.
- 보드의 빈 열은 압축하고 완료 열을 우선 노출한다.
- 타임라인의 `TASK_TRANSITIONED` 같은 내부 이벤트 키를 `Task 상태 변경`처럼 한글로 바꾸고 단계·날짜별로 묶는다.

### P0-3. 회의가 오너 검토에 필요한 한글 의사결정 문서가 되지 못한다

종료된 운영·완료 회의의 Agent 발언은 영어이며 목표·프로젝트·Run UUID가 본문을 차지한다. 회의 목록은 결정 0건을 표시하고 우측 결정 영역도 비어 있다. 승인된 단계인데도 `의미 요약 · 미생성`이 남아 있어 회의가 실제 검토 근거로 신뢰되지 않는다.

개선 방향:

- 오너 기본 화면에는 한글 요약, 합의 내용, 반대 의견, 위험, 미해결 사항, 승인 권고만 보여준다.
- 원문 영어와 UUID는 `기술 근거 보기` 안으로 이동한다.
- 단계 승인 전에 회의 결정 또는 합의 요약이 반드시 생성되도록 계약을 맞춘다.
- 결정 0건인데 단계가 승인된 경우 데이터 정합성 경고와 복구 버튼을 제공한다.

### P0-4. 첫 진입 로딩 중 빈 화면이 노출된다

회사 홈, 직원, 픽셀 오피스는 첫 데스크톱 진입 후 약 1.2초 시점까지 핵심 내용이 비어 있었고, 같은 페이지를 다시 연 모바일 화면에서는 데이터가 표시됐다. 회사 홈 구현은 `busy && !snapshot`일 때 로더나 스켈레톤을 렌더링하지 않는다.

개선 방향:

- 최초 로딩, 새로고침, 빈 데이터, 오류를 서로 다른 상태로 표현한다.
- 페이지별 핵심 구조를 유지하는 스켈레톤과 `회사 현황을 불러오는 중` 상태를 제공한다.
- 로딩 완료 전 `직원 0명`, 빈 오피스처럼 오해할 수 있는 수치를 표시하지 않는다.

## 5. 공통 UI/UX와 디자인 문제

### 정보 구조

사이드바의 10개 운영 메뉴는 모두 같은 위계다. 실제 업무 기준으로 다음처럼 묶는 편이 낫다.

- 회사: 홈, 직원, 목표, 회의
- 실행: 프로젝트, 실행 워크스페이스
- 관찰: 픽셀 오피스, 검색·알림
- 관리자: 운영 플랫폼, 운영 상태

회사를 선택한 뒤 첫 화면은 단순 KPI 모음이 아니라 `오늘의 결정`, `진행 중 목표`, `차단`, `승인 대기`, `최근 완료` 순이어야 한다.

### 언어와 인지 부하

`Project War Room`, `Pixel Office`, `Run`, `Task`, `Agent Backend`, `Delivery Process`, `Diff`가 한글과 섞여 있다. 오너 화면은 한글 용어를 기본으로 하고 기술 용어는 보조로 제공해야 한다. UUID와 내부 이벤트 코드는 상세 근거에서만 노출한다.

### 시각 체계

어두운 공통 셸 안에 목표·회의·알림만 큰 흰 카드로 들어가 화면이 서로 다른 제품처럼 보인다. 카드 내부에 카드가 반복되고 대부분 같은 테두리 강도를 사용해 정보 우선순위가 약하다.

권장 체계:

- 앱 배경, 기본 표면, 상승 표면, 강조 표면의 4단계만 정의
- 핵심 행동은 페이지당 하나의 primary로 제한
- 상태색은 의미에만 사용하고 일반 링크·탭과 분리
- 긴 페이지에는 sticky 단계 요약과 섹션 내비게이션 제공
- 빈 열·빈 패널은 접어서 실제 데이터에 공간을 우선 배분

### 모바일

문서 전체 기준 390px 화면에서 가로 오버플로는 없고 사이드 메뉴도 정상적으로 열렸다. 그러나 다음 세부 문제가 남아 있다.

- 상단 알림 버튼 폭 22px, 여러 버튼·탭 높이 35~36px로 권장 터치 영역 44px보다 작음
- 회사 선택 값과 긴 Agent ID가 잘림
- 회사 카드 이름이 어색한 단위로 줄바꿈됨
- 회의·직원·프로젝트는 가로 밀기 힌트가 있지만 현재 위치와 전체 개수를 알기 어려움
- 픽셀 오피스 캔버스의 이름과 상태가 작은 화면에서 겹치거나 잘림
- 9~10.5px 텍스트가 회의, 목표, 픽셀 오피스에서 확인됨

### 접근성

- Goals에서 이름이 없는 textarea/input 4개와 프로그램 라벨이 없는 프로젝트 연결 select 1개를 확인함
- Project에서 이름이 없는 input 8개, Meetings에서 input/textarea 2개, Pixel Office에서 input 1개를 확인함
- 전역 입력 포커스는 테두리색만 바뀌며 버튼·링크의 일관된 `:focus-visible` 규칙이 없음
- 밝은 목표 영역의 대비는 키보드 사용 여부와 무관하게 읽기 자체를 방해함

## 6. 권장 구현 순서

### 1단계: 신뢰성 복구

1. 공통 상태 프레젠테이션 모델 도입
2. 완료 시 승인·알림·프로젝트·유효 Run 정합화
